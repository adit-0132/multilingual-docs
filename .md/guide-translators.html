rhelpi18n

Translator's Guide

Everything you need to translate an R package's help pages — from installing the tools, to producing a translation module, to publishing it on GitHub so others can install it.

Who this is for You want to translate the documentation (help pages) of an R package into another language. You don't need to know the package's internals — you only edit text files.

1 Prerequisites

  • R 4.4 or newer — check with R.version.string
  • RStudio (recommended — you'll be editing files and running a few commands)
  • git + a GitHub account (for publishing your translation)

2 Install rhelpi18n

rhelpi18n is the engine that builds your translation template and, later, renders translated help.

install.packages("remotes")
remotes::install_github("adit-0132/rhelpi18n@module-skeleton")
Note on the branch Until this work is merged upstream, install from the @module-skeleton branch — that's the one with the skeleton generator. Afterwards it will simply be remotes::install_github("eliocamp/rhelpi18n").

3 Get the package source

You need the package's source code (a folder with a DESCRIPTION file and a man/ folder of .Rd help files) — not the installed version.

# in a terminal
git clone https://github.com/someone/theirpkg.git ~/translate/theirpkg

Or download and unzip the source tarball from CRAN. Either way, you should end up with a folder containing DESCRIPTION and man/.

4 Generate your translation skeleton

Pick your language code (es, fr, zh_CN, pt_BR…) and run:

library(rhelpi18n)

i18n_module_skeleton(
  package_path = "~/translate/theirpkg",     # the SOURCE folder from step 3
  language     = "es",                        # your language code
  module_path  = "~/translate/theirpkg.es"    # where to create your translation
)
This takes a minute It temporarily installs the package in the background so it can see exactly what the help pages look like after installation. That's normal — and the temporary copy is deleted afterwards.

What you get

theirpkg.es/ DESCRIPTION ← says which package + language this translates NAMESPACE R/translation.R ← machinery; don't touch translations/ ← ★ YOUR WORK GOES HERE — one file per help page hello.yaml goodbye.yaml man_original/ ← copies of the original help files, for reference

5 Understand a translation file

Open translations/hello.yaml. It looks like this:

title:
  original: Say hello
  translation: ~
description:
  original: |
    This package was installed under {ISEXPR_0}.
    It always says hello.
  translation: ~
FieldWhat it isDo you edit it?
originalThe English text, exactly as it appears in the help pageNo — never change this
translation~ means "empty". Your translation goes hereYes — this is your job
ifdefAppears on some pages; one entry per platform-specific blockYes — fill each entry

6 The one rule: keep the {ISEXPR_…} placeholders

Some help text changes every time the package is installed — a version number, a date, a generated table. Those parts are replaced by a placeholder like {ISEXPR_0}.

Copy each placeholder into your translation, unchanged. Move it wherever your language needs it, but never translate, renumber, or delete it.

description:
  original: |
    This package was installed under {ISEXPR_0}.
    It always says hello.
  translation: |
    Este paquete se instaló con {ISEXPR_0}.
    Siempre dice hola.
Why this matters The placeholder is filled in with the real, current value when a user opens the help page. Keeping it means your translation keeps working after the package is updated or reinstalled.

Platform-specific blocks (ifdef)

Some pages show different text on Windows vs. Mac/Linux. Those look like this — translate each numbered branch:

description:
  original: |
    Text below appears on everything except Windows:
    {ISEXPR_0}Text after always appears.
  translation: |
    El texto de abajo aparece en todo excepto Windows:
    {ISEXPR_0}El texto posterior siempre aparece.
  ifdef:
    '0': ESTA LÍNEA SE MUESTRA EN UNIX Y MACOS (NO WINDOWS).
Rare: literal braces If the original shows {{ISEXPR_0}} with double braces, that's real text the author wrote — not a placeholder. Keep the double braces as-is in your translation.

7 Test your translation locally

# 1. install your translation module
install.packages("~/translate/theirpkg.es", repos = NULL, type = "source")

# 2. RESTART R  (Session → Restart R, or Ctrl+Shift+F10)

# 3. switch language and open a help page
library(rhelpi18n)
library(theirpkg)
Sys.setenv(LANGUAGE = "es")
?hello

The help pane should show your Spanish text, with the real value filled into the placeholder. To compare, set Sys.setenv(LANGUAGE = "en") and open it again.

Always restart R after installing If you install a package and open its help in the same session, R may show a stale version. Restart first — it fixes almost every "my change didn't show up" problem.

8 Publish it on GitHub

Your translation module is just a small R package. Put it on GitHub so others can install it.

# in a terminal, from inside your module folder
cd ~/translate/theirpkg.es

git init
git add .
git commit -m "Spanish (es) translation for theirpkg"

# create the repo and push (needs the GitHub CLI)
gh repo create theirpkg.es --public --source=. --push

No GitHub CLI? Create an empty repo on github.com, then:

git remote add origin https://github.com/YOURNAME/theirpkg.es.git
git branch -M main
git push -u origin main

9 Tell people how to install it

Put this in your repo's README:

remotes::install_github("adit-0132/rhelpi18n@dynamic-sexpr")  # the engine
remotes::install_github("YOURNAME/theirpkg.es")               # this translation

# then restart R:
Sys.setenv(LANGUAGE = "es")
?hello

Keeping your translation up to date

When the package you translated releases a new version, its help text may have changed. Re-run step 4 into a new folder and compare: sections whose original changed need re-translating; the rest you can copy across.

Why man_original/ exists It stores the exact help files your translation was made from, so you (or a tool) can diff them against a newer version to see precisely what changed.

Troubleshooting

SymptomFix
Help still shows EnglishRestart R. Then check Sys.getenv("LANGUAGE") and that your module is installed.
One section is English, others translatedThat section's translation: is still ~, or its original no longer matches the installed package (package was updated).
Placeholder shows as literal {ISEXPR_0}The text around it changed. Regenerate the skeleton and re-copy your translation.
i18n_module_skeleton errorsCheck package_path points at a folder with DESCRIPTION + man/, and that the package installs cleanly on its own.

Use underscores, not hyphens Write zh_CN, not zh-CN. R's language setting uses underscores, and the match won't be found otherwise. (The package name gets dots automatically — that's expected.)
