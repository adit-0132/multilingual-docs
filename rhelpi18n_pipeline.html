the core idea
Translations live in pkg.es/inst/translations/*.yaml, edited by humans. rhelpi18n's only job at runtime is to match the installed (baked) help against each stored original and swap in the translation. The dynamic-\Sexpr work changes just one thing: the stored original becomes a scaffold with {ISEXPR_i} tokens, so a human's translation keeps matching after every reinstall.

N1 — the version pin (in progress)
Translates: pkg (== x.y.z) records the exact version the scaffolds were built for. Today get_translation_modules() parses this but ignores the version. Enforcing it (the DEC node) is the fix for the "documentation drifted next to a placeholder" concern: a doc change ships a new version → pin fails → original is shown, never a wrong translation.

match_and_fill — the algorithm Split stored on {ISEXPR_i} → N+1 literal anchors. Pin the first/last with startsWith/endsWith, locate interior anchors sequentially (fixed-string, spans newlines). Gaps = captured live values → substituted into the translation's tokens.
#ifdef branch table A per-token ifdef side-table maps a token to its translated branch. If the captured value is the active branch (not R's not active marker) it is replaced by the stored translation; the inactive branch keeps its invisible marker.
fail-safe (C4) Any non-match returns live unchanged — the caller uses the result unconditionally. A stale scaffold degrades to the original language, never a wrong render.
N2 — escaping (planned) Docs may literally contain {ISEXPR_0}. Plan: double-brace escape {{…}} in anchors at build time, match only single-brace tokens, un-double after substitution.

the fix vs today
Today i18n_translation_templates writes original = raw source flatten, which for an install/build \Sexpr still contains the live node text — it can never equal the baked runtime string, so those sections silently stay English. Folding detect_scaffolds in makes original a scaffold that matches the installed binary. The baked tree can come from Rd_db (installed) or prepare_Rd (from source).

what the human touches Only the translation: fields. The original: scaffold is generated — never hand-authored. The one rule: leave {ISEXPR_i} tokens exactly where they are.
what the human does NOT need They do not install or run the package being translated (that was Elio's confusion, C2). Only the maintainer needs the package once, to generate scaffolds.
maintenance loop On a version bump, regenerate scaffolds; sections whose scaffold changed are flagged for re-translation. A changed install value raises no false flag — only changed static prose does.

\Sexpr stages & where values are stored

build/install bake to text in the .rdb; render stays live; all feed the scaffold as token or static anchor

build — recoverable R CMD build does not bake it; the source tarball keeps it live. It bakes at install, so source↔baked diff can tokenise it.
install — the problem case Bakes a volatile value (timestamps, versions, weekdays) into pkg.rdb every install. This is exactly what the {ISEXPR_i} token absorbs.
render — needs no source Stays a live node in the .rdb, re-evaluated at display. Detectable from the binary alone; kept verbatim inside anchors.
where it lives on disk <lib>/pkg/help/pkg.rdb (gzip lazy-load db) + pkg.rdx (topic → offset). Read via tools::Rd_db() and by the .getHelpFile shim.

Translation module anatomy

A normal R package whose translations object is the human-edited YAML, keyed by topic

pkg.es/ DESCRIPTION ← Translates: pkg (== x.y.z) · Language: es NAMESPACE ← export(translations) R/ translation.R ← translations <- read inst/translations/*.yaml at load inst/translations/ greet.yaml ← section: { original: scaffold, translation:, ifdef: } read_xml.yaml man_original/ ← copy of source Rd (for re-diff / staleness on version bump)
one YAML section
original: "You installed the package on {ISEXPR_0}"  ·  translation: "Instalaste el paquete un {ISEXPR_0}". The runtime matches the live baked string against original, captures the value into {ISEXPR_0}, and substitutes it into translation. man_original/ is what makes the maintenance re-diff possible.
