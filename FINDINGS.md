which the existing `rhelpi18n` silently mishandles.

### Direction: proposal vs. current instructions (important reconciliation)

The PDF proposal describes building a **new package** that intercepts
`utils:::.getHelpFile()`. The current `CLAUDE.md` + `R notes` override this:

> Work directly in the `rhelpi18n` repo. Do not start a new package from scratch
> unless explicitly instructed.

So the live plan is to **extend `rhelpi18n` in place**, not greenfield a new
package. The proposal's architecture (shim at `.getHelpFile`, parse-tree-level
work, placeholder tokens for render-stage `\Sexpr`, module creation from the
*installed* Rd database) is still the intended design ‚Äî it's just landing inside
`rhelpi18n`. Treat the proposal as design rationale, not a literal task list.

## 2. Rd macro stages (the core mental model)

R help is authored as `.Rd`, compiled into a binary `.rdb` database at install
time. `\Sexpr[...]{ <R code> }` embeds R, evaluated at one of three stages:

| Stage | Runs during | Visible in installed `.rdb`? | Implication |
|---|---|---|---|
| `stage=build` | `R CMD build` (tarball) | No ‚Äî gone before install | Out of scope. |
| `stage=install` | `R CMD INSTALL` | **Result baked in as plain TEXT** | No longer a live node; reads as ordinary text. |
| `stage=render` | every `?topic` call | **Yes ‚Äî still a live `\Sexpr` node** | Must be preserved verbatim and re-evaluated at help time. |

Detecting a live render-stage node from the compiled db:

```r
is_render_sexpr <- function(node) {
  identical(attr(node, "Rd_tag"), "\\Sexpr") &&
    isTRUE(grepl("stage=render", attr(node, "Rd_option")))
}
```

**`#ifdef` caveat.** Platform conditionals (`#ifdef windows ‚Ä¶ #endif`) are
resolved *after build macros but before install macros*. An installed `.rdb`
therefore only contains text for the platform it was installed on. Documented
limitation; not in scope to fix.

**CRAN survey baseline (from `R notes`).** Grepping CRAN for `\Sexpr[`:
- **91** packages use `\Sexpr` at all.
- **48** remain after dropping `stage=install` and `stage=build` (the
  render-stage candidates ‚Äî the selected lines 132‚Äì179 in `R notes`).
- Skim impression: most render-stage `\Sexpr` are *formatting, colours, and
  method listings* ‚Äî much of it may not need translating.

## 3. `rhelpi18n` codebase walkthrough (`main` branch)

Package metadata: `Depends: R (>= 4.3.0), utils`; `Imports: pkgload, yaml,
whisker`. Exports: `i18n_module_create`, `i18n_translation_templates`,
`rd_flat_write`, `read_translations`, `print.Rd_flat`.

### The runtime pipeline

```
help("topic")
  ‚îî‚îÄ utils:::.getHelpFile(file)            ‚Üê patched in .onLoad (zzzz.R)
        ‚îî‚îÄ on.exit ‚Üí .translateHelpFile(rd, pkgname, file)   (getHelpFile-shim.R)
              ‚îú‚îÄ get_translation_modules(pkg, language)       (get-translation-modules.R)
              ‚îú‚îÄ translations <- module's `translations`[[basename(file)]]
              ‚îî‚îÄ rd_translate(rd, translations)               (rd-translate.R)
                    ‚îú‚îÄ rd_flatten(rd)        ‚Üí flat list, one entry per section  (rd-flatten.R)
                    ‚îú‚îÄ translate(flat, translation)   ‚Üí swap originals for translations
                    ‚îî‚îÄ rd_unflatten(translated)  ‚Üí write Rd text to tempfile, re-parse  (rd-unflatten.R)
```

### File-by-file

- **`zzzz.R` (`.onLoad`)** ‚Äî surgically rewrites `utils:::.getHelpFile`: injects
  `on.exit(return(.translateHelpFile(returnValue(), pkgname, file)))` as the
  first statement, then `unlock/assign/lockBinding`. This is the single
  interception point all help viewers (console, RStudio, browser) converge on.

- **`getHelpFile-shim.R` (`.translateHelpFile`)** ‚Äî looks up a translation
  module for `pkgname` + `LANGUAGE`, pulls the module's `translations` list,
  indexes by the Rd file basename, and calls `rd_translate`. Returns `rd`
  unchanged if no module / no translation.

- **`get-translation-modules.R`** ‚Äî scans `installed.packages()` for the custom
  `Translates` and `Language` DESCRIPTION fields, filters by language
  (`resolve_lang`) and target package.

- **`module-create.R` (`i18n_module_create`)** ‚Äî scaffolds a translation-module
  package: copies the skeleton, fills DESCRIPTION via `whisker`, then builds YAML
  templates. **‚öÝ Reads source `man/*.Rd` directly** (`list.files(.../man)` +
  `tools::loadPkgRdMacros`) ‚Äî i.e. *pre-install*, so install-stage `\Sexpr` are
  unevaluated raw expressions here. This is root cause #1 (below).

- **`rd-flatten.R`** ‚Äî the heart. `rd_flatten()` turns the nested Rd into a flat
  named list keyed by section tag (`title`, `description`, `\section{Methods}`,
  ‚Ä¶). `arguments` is handled specially (per-`\item` name/description). The
  workhorse `to_text()` collapses any node to a string; **for any tag starting
  with `\\` it does `as.character(setRd(x), deparse = TRUE)`** ‚Äî so a live
  `\Sexpr` node is serialised to its *raw expression string*. This is where
  structural info is lost (root cause #2).

- **`rd-translate.R` (`translate`)** ‚Äî for each section, the **version check**
  `original[[section]]$original == translation[[section]]$original` gates whether
  the stored translation is applied. `examples`/`title` get a `<details>` HTML
  toggle wrapping translated + original; others are replaced outright. If the
  check fails, it silently keeps the original.

- **`rd-unflatten.R`** ‚Äî `list2rdtext()` re-serialises the flat list to `\tag{‚Ä¶}`
  Rd text, writes a tempfile, and `tools::parse_Rd`s it back. `section2char()`
  writes `\item{name}{value}` for arguments. **Arg names like `...` are not
  escaped** before re-parse (root cause #3).

- **`rd-flat-write.R` / `read-translations.R` / skeleton `R/translation.R`** ‚Äî
  YAML I/O. `untranslatable` fields (`alias`, `name`, `keyword`, `concept`,
  `usage`) are stripped from templates and recorded as an attribute. The module
  builds its `translations` list at load time from `translations/*.yaml`.

- **`rd-flatten-po.R`** ‚Äî parallel experiment emitting gettext `.po`
  (`msgctxt`/`msgid`/`msgstr`) instead of YAML. Maps to the `po-files` remote
  branch. Not the primary path.

- **`resolve-lang.R`** ‚Äî ISO language matching (`es_AR` ‚Üí `es`), exact match
  preferred, root-language fallback. (Issue #9.)

- **`path-component.R`** ‚Äî dead helper, kept "just in case" (author's note).

### YAML schema (from `notes/mean.es.yaml`, the worked example)

```yaml
title:
  original: Arithmetic Mean
  translation: Media Aritm√©tica
arguments:
  x:
    original: |- ...
    translation: |- ...
  \dots{}:                       # note: '...' serialised as \dots{}
    original: further arguments passed to or from other methods.
    translation: otros argumentos, ...
value:
  original: |2 ...
  translation: ~                 # untranslated ‚Üí stays original
```

One YAML file per Rd topic; each section has `original` + `translation`
(`~`/null = not translated ‚Üí graceful fallback to original). Backward-compat
rule from `CLAUDE.md`: files lacking new fields must default to fully static.

## 4. The two branches: `main` vs `Rdfiles`

`Rdfiles` is exactly **one commit ahead** of `main` (`c7a2b3e "Use RD files"`).
It is an **experimental alternative interception strategy**, not a feature
addition. Diff is confined to `zzzz.R`, `getHelpFile-shim.R`,
`get-translation-modules.R` (plus a roxygen/version bump).

| Aspect | `main` (in-memory translation) | `Rdfiles` (serve translated files) |
|---|---|---|
| Patches | `utils:::.getHelpFile` | `base::find.package` |
| Mechanism | After Rd is fetched, translate the parse tree in memory (`rd_translate`) and return it | Prepend the translation module's dir to the package paths `find.package` returns, so the help system *discovers the module's own Rd files* |
| New helper | ‚Äî | `add_translation_modues()` (sic ‚Äî typo) prepends module paths; `parse_deps` inlined to drop the `pkgload` dep |
| Maturity | The proposal's chosen path; complete enough to run | Sketch: one commit, typo, relies on the module physically shipping translated `.Rd`/`.rdb` |
| Trade-off | Full control over the tree (needed for `\Sexpr` placeholders); must re-implement rendering hooks | Reuses R's native rendering for free; but you must *pre-render* translated Rd files and dynamic `\Sexpr` semantics get murkier |

**Read:** `main` is the active line and matches the proposal (parse-tree-level
work is what the `\Sexpr` placeholder design needs). `Rdfiles` parks an idea
worth discussing with mentors but isn't where the dynamic-content work should be
built. Other remote branches: `po-files` (gettext route), `gh-pages` (site).

## 5. Root causes (proposal ¬ß5.3, confirmed against code)

| # | Problem | Location | Root cause |
|---|---|---|---|
| 1 | Install-stage `\Sexpr` never translates | `module-create.R:42`, `rd-translate.R:27` | Module built from **source `man/`** (pre-install) ‚Üí YAML `original` stores the *raw* expr; at help time the compiled Rd has the *evaluated text*. Version check never matches ‚Üí translation silently dropped. **Fix:** build modules from `tools::Rd_db()` of the *installed* package. |
| 2 | Render-stage `\Sexpr` lost/corrupted | `rd-flatten.R` `to_text()` | Live node deparsed to a raw string inside the section's `original`; no placeholder, no side-store. Translator can silently break it. **Fix:** replace each render node with an opaque token `{SEXPR_0}` + side list of original nodes; restore before re-parse. |
| 3 | `...` arg names malformed on round-trip | `rd-unflatten.R` `section2char()` | Special Rd chars in arg names not escaped before writing the tempfile. **Fix:** escape in `section2char()`. |

> Nuance on #1 vs render-stage: for **render-stage**, both the source and the
> compiled Rd still contain a live `\Sexpr` node that deparses to the *same*
> string, so the version check actually *passes*. Render-stage breakage is purely
> #2 (flatten/translator fragility + re-parse), not the version-check mismatch
> that kills install-stage.

Design stance (proposal ¬ß6): don't patch text-level symptoms ‚Äî **work at the Rd
parse-tree level throughout**, only collapsing to text for sections confirmed
fully static.

## 6. `\Sexpr` survey ‚Äî methodology & first result

Reusable script committed at repo root: [sexpr-survey.R](sexpr-survey.R).
It reads the **compiled** db (`tools::Rd_db(pkg)`), recursively collects every
node with `Rd_tag == "\\Sexpr"`, and prints `topic / Rd_option / expression`.

```sh
Rscript sexpr-survey.R dplyr     # or any installed candidate
```

Installed candidates from the selected lines (others need `install.packages`):
`dplyr, pkgdown, DBI, generics, vroom, xml2, scales`.

### Chosen first target: `dplyr` (38 render-stage uses; recommended in CLAUDE.md)

**Every one of dplyr's 38 live `\Sexpr` nodes is identical in shape:**

- `Rd_option`: **`stage=render,results=rd`** (all 38)
- expression: **`dplyr:::methods_rd("<verb>")`** ‚Äî `arrange`, `mutate`,
  `filter`, `left_join`, `slice_*`, ‚Ä¶ (21 topics)

`methods_rd()` returns either `"no methods found"` (current env ‚Äî backends
absent) or, when backends like dbplyr/dtplyr are loaded, an Rd-markup string of
the form `dbplyr (\link[dbplyr]{...}), dtplyr (\link[dtplyr]{...})` ‚Äî a
**dynamically-computed list of S3 method dispatch links**. `results=rd` means
that string is re-parsed as Rd.

**Why this is the perfect first case.** Each sits inside a `\section{Methods}`:

```
This function is a \strong{generic}, which means that packages can provide
implementations (methods) for other classes. ...
The following methods are currently available in loaded packages:
\Sexpr[stage=render,results=rd]{dplyr:::methods_rd("arrange")}.
```

The section is **translatable scaffold prose + a live render node that must stay
verbatim**. Running the *current* `main` flattener (`rd_flatten`) on the
compiled `arrange.Rd` yields, as the YAML `original` a translator would edit:

```
This function is a \strong{generic}, which means that packages can provide
implementations (methods) for other classes. See the documentation of
individual methods for extra arguments and differences in behaviour.

The following methods are currently available in loaded packages:
\Sexpr[stage=render,results=rd]{dplyr:::methods_rd("arrange")}.
```

‚Üí The translator is handed the raw `\Sexpr[...]{...}` (and `\strong{}`) inline
and must reproduce them byte-for-byte. **This is root cause #2 made concrete** ‚Äî
and precisely what the `{SEXPR_0}` placeholder design removes: translator sees
clean prose + an opaque token; the live node is restored before re-parse and
re-evaluated at render time.

**Translation verdict for dplyr:** the *Sexpr output* (method links) should be
preserved, **not** translated; only the surrounding scaffold prose is
translatable. Confirms the `R notes` hypothesis that method-listing `\Sexpr` are
structural, not prose.

### 8.2 Methodology correction (matters for candidate selection)

**The default `\Sexpr` stage is `install`, not render** (WRE / `tools::parse_Rd`).
The `R notes` filter #2 (`grep -v stage=install | grep -v stage=build`) only drops
*explicit* `stage=install`, so it **wrongly retains every no-stage `\Sexpr`**
(which is really install-stage). Reproducing that filter on the CSV keeps **67**
packages, but only **35** have a genuine explicit `stage=render`. The other 32
are default-install masquerading as render candidates, e.g.: `knitr`, `Hmisc`,
`GGally`, `collapse`, `dbplyr`, `katex`, `googleVis`, `ggthemes`, `xml2`,
`Rdpack`, `geostatsp`, `HH`, `matlab`, `prefmod`, `this.path`, ‚Ä¶

> Practical filter going forward: select on `stage=render` **explicitly**; treat
> everything else (including no-stage) as install/build = baked text = not a live
> node. The real working set is **35 packages / 1477 render occurrences**.

### 8.3 What render-stage `\Sexpr` actually generates (by kind)

Crude classification of the 1477 render occurrences:

| Category | Occ. | Translate? | Typical generator |
|---|---|---|---|
| colour / CSS / icon / status formatting | 567 | **No** ‚Äî visual | `bs4Dash`, `shinydashboardPlus` swatches |
| "other" (term metadata / engine lists / generated bodies) | 442 | **Mixed** ‚Äî see 8.4 | `ergm:::.formatTermKeywords`, `parsnip:::make_engine_list`, `stevedore:::generate_help` |
| method / aesthetic dispatch listing | 226 | **No** ‚Äî structural links | `dplyr:::methods_rd`, `rd_aesthetics`, `*_rd` |
| lifecycle badge | 183 | **No** (standardised label) | `lifecycle::badge("experimental")` |
| math / LaTeX rendering | 49 | **No** | `katex`, formula renderers |
| version / date / option | 10 | **No** (dynamic value) | `packageVersion`, `Sys.*` |

**Headline insight:** the bulk of render-stage `\Sexpr` is *structural or visual*
and must be **preserved verbatim, not translated** ‚Äî which is exactly the
placeholder/`{SEXPR_0}` case, not the "translate this text" case. Genuine
translatable prose generated *by* a `\Sexpr` is rare; the value is almost always
in the **static scaffold prose around** the node. This validates the proposal's
design priority (preserve the node, translate the scaffold).

### 8.4 Notable structural patterns (these drive the placeholder design)

- **Scaffold + node, node preserved** (the common, tractable case):
  `dplyr` / `parsnip` ‚Äî `\section{Methods}` or See-Also with prose intro +
  `\Sexpr{methods_rd(...)}` / `make_engine_list(...)`. Translate the intro,
  keep the node as a token. **This is the primary pattern to support.**
- **Inline mid-sentence node** ‚Äî `lintr`:
  `‚Ä¶\Sexpr[stage=render,results=rd]{lintr:::regexes_rd}. A name should ‚Ä¶`
  The token lands *inside* a sentence with translatable text on both sides.
  Tests placeholder positioning, not just whole-section replacement.
- **Per-format conditional node** ‚Äî `ergm` (and the whole ergm family):
  `\if{html}{\Sexpr{...formatProposalsHtml...}}`,
  `\if{text}{\Sexpr{...formatProposalsText...}}`,
  `\if{latex}{\Sexpr{...formatProposalsLatex...}}` ‚Äî three sibling render nodes,
  one per output format, plus `\Sexpr{.formatTermKeywords(...)}` subsections.
  Stress-tests the flattener against `\if{}` wrappers + multiple nodes per section.
- **Whole-body generation** ‚Äî `stevedore:::generate_help(...)`: essentially the
  *entire* help page is produced at render time. There is little/no static
  scaffold to translate. Edge case ‚Äî likely "document as unsupported / preserve".
- **Self-test fixture, not real docs** ‚Äî `pkgdown`'s 2 render nodes are literally
  `\Sexpr[stage=render,results=rd]{"sexpr"}` (the string `"sexpr"`). **Exclude.**

### 8.5 Recommended test targets (ranked, "relevant to our cause")

"Relevant" = a render `\Sexpr` embedded in or beside **translatable prose**, so it
actually exercises the scaffold-translate + node-preserve pipeline.

| # | Package | Render n | Pattern | Why test it | Status |
|---|---|---|---|---|---|
| 1 | **dplyr** | 38 | methods_rd in `\section{Methods}` | Clean canonical scaffold+node; popular; already characterised (¬ß6) | **installed** |
| 2 | **lintr** | 1 | inline mid-sentence token | Smallest reproducer of in-prose placeholder positioning | install |
| 3 | **parsnip** | 60 | engine list + seealso list + intro prose | Uniform, high-volume, well-known (tidymodels); strong scaffold-translate case | install |
| 4 | **ergm** | 299 | `\if{html/text/latex}{\Sexpr}` + term-keyword subsections | Heaviest user; stress-tests `\if{}` + multi-node sections (heavy `statnet` deps) | install |
| 5 | **stevedore** | 22 | whole-body `generate_help()` | The "no static scaffold" edge case ‚Äî defines a limit of the approach | install |

**Baseline / regression ("must-not-corrupt", low translation value):**
- `vroom` (4, lifecycle badges) ‚Äî **installed**, trivial uniform baseline.
- `DBI` (47) / `generics` (39) ‚Äî **installed**, pure method listings; good
  no-op round-trip targets.

**Deliberately deprioritised (volume trap):**
- `bs4Dash` (514) and `shinydashboardPlus` (53) ‚Äî *most* `\Sexpr` of anyone, but
  100% colour/CSS/status visuals with **zero translatable text**. High count ‚âÝ

Start with **dplyr** (done, ¬ß6) ‚Üí add **lintr** (inline) and **parsnip**
(scaffold-heavy) as the placeholder-mechanism test bed ‚Üí keep **ergm** as the
hard structural stress test ‚Üí use **vroom/DBI** as no-op round-trip guards.
`Rscript sexpr-survey.R <pkg>` still works to dump live nodes once a package is
installed.

## 9. The `gsocproposal` macro showcase (this package)

Follow-up session. The demo package's single help page,
[`man/submit_proposal.Rd`](man/submit_proposal.Rd), was rebuilt by hand into an
exhaustive, **installs-clean (zero warnings)** showcase of every Rd construct, so
we have a controlled fixture to test the translation pipeline against. Companion
user-macro file: [`man/macros/gsoc-macros.Rd`](man/macros/gsoc-macros.Rd).

### 9.1 What it exercises

| Group | Constructs |
|---|---|
| `\Sexpr` stages | `stage=install` (deadline, baked to text), `stage=render` (live message), `stage=build`, default (= install) |
| `\Sexpr` results | `results=text`, `results=verbatim` (`[1] 2`), `results=rd` (emits Rd, re-parsed), `results=hide`, `echo=TRUE` (echoes source) |
| Conditionals | `#ifdef windows ‚Ä¶ #endif`, `#ifndef windows ‚Ä¶ #endif` (correct line-leading `#`, nested in a section) |
| User macros | `\gsoc`, `\pkgenv` (0-arg), `\deadlineNote{#1}` (1-arg) from `man/macros/` |
| Format conditionals | `\if{html}{\out{‚Ä¶}}`, `\if{text}{‚Ä¶}`, `\ifelse{html}{}{}` |
| Inline markup | `\emph \strong \bold \code \var \pkg \file \option \env \command \kbd \sQuote \dQuote \dfn \acronym \cite \verb \samp \R \enc`, escaped `\%` |
| Lists / table / math | `\itemize \enumerate \describe`, `\tabular`/`\tab`/`\cr`, `\eqn \deqn` |
| Links | `\link`, `\link[pkg]`, `\link[=target]`, `\url`, `\href` |
| Examples wrappers | `\dontrun \donttest \dontshow`; `\method` in `\usage`; `\dots` in `\arguments` |

### 9.2 Verified rendering behaviour (Rd2txt + Rd2HTML, render stage)

- **install** `\Sexpr` ‚Üí `Deadline: 2025-03-31` baked into the `.rdb` as text.
- **render** `\Sexpr` ‚Üí recomputed live (`The deadline has passed‚Ä¶`, today's date).
- **build** `\Sexpr` ‚Üí evaluated when installing from a source *dir* too.
- `#ifndef windows` branch shown, `#ifdef windows` branch dropped (Linux box).
- `\if{html}{\out{...}}` emits raw `<strong>` in HTML, suppressed in text.
- `\deadlineNote{2025-03-31}` ‚Üí "The deadline is **2025-03-31**".

### 9.3 Escaping gotcha ‚Äî `results=rd` needs **four** backslashes

A `results=rd` Sexpr whose R code returns Rd markup must be written with `\\\\` in
the `.Rd` source: `\Sexpr[results=rd]{ "\\\\emph{x}" }`. Two layers halve it ‚Äî the
Rd-database deparse once, R string parsing again ‚Äî leaving valid `\emph{x}` to
re-parse. Two backslashes throws `'\e' is an unrecognized escape` at render.

### 9.4 Live-vs-baked node counts (the project premise, on our own fixture)

After install, `tools::Rd_db("gsocproposal")[["submit_proposal.Rd"]]` keeps
**6 live `\Sexpr` nodes ‚Äî all `stage=render`**; the install/build ones are gone
(baked to text) and both `#ifdef` nodes are resolved away. Exactly the
live-vs-baked split `rhelpi18n` must handle (¬ß2, root cause #1).

## 10. How `tools::parse_Rd()` represents `\Sexpr` / `#ifdef` (tasks 2 & 5)

Reusable inspector at repo root: [`../rd-inspect.R`](../rd-inspect.R) ‚Äî parses the
source `.Rd` *and* reads the installed db, then lists every `\Sexpr`,
`#ifdef`/`#ifndef`, and `USERMACRO` node with its options.

### 10.1 The parser identifies them natively (answer to task 2: **yes**)

Each construct gets a dedicated `Rd_tag`, so detection is an attribute check ‚Äî no
string scraping:

| Construct | Representation in the parse tree |
|---|---|
| `\Sexpr` | `Rd_tag = "\\Sexpr"`; stage/results in **`attr(node, "Rd_option")`** (e.g. `results=text,stage=render`); child `RCODE` node holds the R code |
| `#ifdef` / `#ifndef` | `Rd_tag = "#ifdef"` / `"#ifndef"`; a **two-argument** node ‚Äî `node[[1]]` = condition (` windows`), `node[[2]]` = conditional body (matches parseRd ¬ß2.3) |
| user macro | `Rd_tag = "USERMACRO"` with **`attr(node, "macro")` = the macro name** (e.g. `"\\gsoc"`) |
| format conditionals | `\\if`, `\\ifelse`, `\\out` |

`is_render_sexpr()` from ¬ß2 is therefore the whole detector. The parser also
exposes `attr(node, "srcref")` (line/col positions) for every node.

### 10.2 Source parse vs installed db ‚Äî the decisive contrast

Same file, two views (via `../rd-inspect.R`):

| | `parse_Rd(man/submit_proposal.Rd)` ‚Äî **source** | `Rd_db("gsocproposal")` ‚Äî **installed** |
|---|---|---|
| `\Sexpr` nodes | **9** (build + install + render + default) | **6** ‚Äî render only |
| `#ifdef` / `#ifndef` | **2** (both branches, unevaluated) | **0** ‚Äî resolved to one platform's plain text |

This is root cause #1 demonstrated on our own fixture: build/install Sexpr bake to
text and `#ifdef` collapses at install, so **module creation must read `Rd_db()`,
not source `man/`**.

### 10.3 `rdviewer` (task 5): useful for structure, blind to `Rd_option`

[`rdviewer`](../rdviewer) overrides `print.Rd` to collapse the deep nested list
into a readable `<tag>` tree ‚Äî good for *seeing nesting* at a glance. Two caveats
for our use:

1. **It does not print `Rd_option`** ‚Äî so a Sexpr's `stage`/`results` and a
   `\link`'s target are invisible in its tree. For us that's the key attribute.
2. **S3 `print.Rd` clashes with `tools`** ‚Äî whichever package loads last wins
   (observed `tools` overwriting `rdviewer` mid-session), so the tree is finicky.

`../rd-inspect.R` includes a small **option-aware** tree that adds `[Rd_option=‚Ä¶]`,
producing what we actually need:

```
‚îî‚îÄ[3]<\Sexpr>  [Rd_option=results=text,stage=render]
    ‚îî‚îÄ[1]<RCODE> "format(Sys.Date())"
‚îî‚îÄ[5]<#ifdef>
    ‚îî‚îÄ[1]<NULL> ‚îî‚îÄ[1]<TEXT> " windows\n"            # arg1 = condition
    ‚îî‚îÄ[2]<NULL> ‚îî‚îÄ[1]<TEXT> "A Windows-only note.\n" # arg2 = body
```

Natural small upstream contribution: teach rdviewer to print `Rd_option`.

### 10.4 Lead for the `\Mark_Sexpr` trace idea 

Suggested a `\newcommand` marker to **keep a trace of text substituted
by a `\Sexpr`**. The mechanism that makes this viable is now confirmed:
`USERMACRO` nodes carry **`attr(node, "macro")` = the macro name** and **survive
into the installed db** (`str()` shows `attr(*, "macro")= chr "\\gsoc"`). So a
`\newcommand`-based wrapper around `\Sexpr` would leave a *findable marker* in the
compiled tree ‚Äî exactly the trace requested. Caveat to test next: the USERMACRO
node stores a *flattened* copy of the macro body (e.g. `"The deadline is
\strong{#1}.2025-03-31"`, `#1` unsubstituted), so a prototype must confirm the
inner `\Sexpr` stays **live** alongside the marker, not flattened.

## 11. Session 3 ‚Äî minimal fixtures, trace-wrapping, and a working `es` translation

### 11.1 Demo fixtures (one construct per page, `?topic`-callable)
`man/`: `sexpr_build`, `sexpr_install`, `sexpr_render`, `ifdef`, `ifndef`,
`usermacro`, `test` (scratch), `greet` (translation demo). Inspectors:
`walk_rdb_tree(topic)` (exported; reads the installed `.rdb`); sourceable live
walkers in `scripts/walk_rd_file.R` ‚Üí `walk_rd_file()` (source tree) and
`walk_rdb_file()` (baked tree). Full per-construct trees: `docs/rdb-tree-report.md`.

### 11.2 What survives into the compiled `Rd_db()` tree
| Construct | In `Rd_db()` | Self-identifying? |
|---|---|---|
| build / install `\Sexpr` | plain `TEXT` (evaluated) | ‚ùå looks like prose |
| render `\Sexpr` | live `\Sexpr` + `Rd_option` | ‚úÖ |
| `#ifdef` **inactive** | `COMMENT: "#ifdef <os> not active"` | ‚úÖ trace left |
| conditional **active** | plain `TEXT` | ‚ùå |
| user macro | `USERMACRO` + `attr "macro"` | ‚úÖ tagged + named |

### 11.3 Live baked loop (no reinstall/restart)
`walk_rdb_file(path)` = `parse_Rd()` ‚Üí `tools:::prepare_Rd(stages=c("build","install"))`,
reproducing the `.rdb` tree **in memory from a source file** ‚Äî output identical to
`Rd_db()`. `defines="windows"` previews the Windows `#ifdef` branch from Linux.
(Uses an internal fn; verified on R 4.5.)

### 11.4 Wrap-to-trace (`\Mark_Sexpr`) ‚Äî prototyped (extends ¬ß10.4)
- `\newcommand` markers leave `USERMACRO` nodes (`attr "macro"`), **survive
  install**, are **invisible** in output ‚Üí distinguish Sexpr-text from authored
  prose. Empty open/close marker macros give a true start/end bracket.
- **Underscore names break**: `\Sexpr_expanded` parses as built-in `\Sexpr` +
  `_expanded`. Use letters only (`\SexprExpanded`).
- **Install‚Üírender emit** (`\Sexpr[results=rd,stage=install]{‚Ä¶}` returning a
  render `\Sexpr`): leaves a **live render node (trace) in the `.rdb`**, but R
  **does not re-evaluate** it at help time ‚Üí renders as literal source. Good as a
  *marker*, not a value-renderer. Needs **4 backslashes** (`\\\\Sexpr`).
- Nested `\Sexpr{\Sexpr{‚Ä¶}}` is **invalid** (outer `{}` must be R code).
- Clean author path stays `\newcommand{\translate}{\Sexpr[results=text,stage=render]{#1}}`.

### 11.5 No CRAN package uses all three stages
`sexpr-usage.csv`: zero packages combine build+install+render ‚Äî they specialize.
So **`gsocproposal` is the only all-three fixture** (justifies the demo package).

### 11.6 Working `es` translation
- Module **`gsocproposal.es/`**: DESCRIPTION `Translates: gsocproposal (== 0.1.0)`,
  `Language: es`; YAML in `inst/translations/greet.yaml`; `translations` object
  built via `system.file`.
- YAML generated from the **compiled `Rd_db` flatten** (originals match help-time ‚Üí
  sidesteps root-cause #1); dummy Spanish; the `\Sexpr` kept **verbatim** in `details`.
- **Discovery = two DESCRIPTION fields.** rhelpi18n scans
  `installed.packages(fields=c("Translates","Language"))`. Definitive check:
  `rhelpi18n:::get_translation_modules("gsocproposal","es")` ‚Üí `"gsocproposal.es"`.
- **Demo:**
  ```r
  library(rhelpi18n); library(gsocproposal)
  Sys.setenv(LANGUAGE = "es"); ?greet     # Spanish
  ```
  ‚Üí Spanish title/description/value; the `details` **render Sexpr evaluates live
  inside the Spanish text** (`La fecha actual es 2026-06-12‚Ä¶`); the install-baked
  `R 4.5.3` is frozen into the translated string. `LANGUAGE="en"` ‚Üí English.
- **Updates ¬ß10.5:** render `\Sexpr` is *not* doomed to come back untranslated ‚Äî
  kept verbatim in the translation it re-parses and re-evaluates correctly. The
  hard part is purely *not corrupting* the node (the placeholder problem, ¬ß5 #2).
- Source dir ‚âÝ runtime: a translation edit needs `R CMD INSTALL gsocproposal.es`
  to land (same source-vs-installed split as the walkers).

### 11.7 Gotchas (demo survival kit)
- **Restart R after any (re)install** of the package *or* the module. Stale
  `.getHelpFile` patch + module namespace + RStudio help cache ‚Üí help-pane *500*.
  A clean session fixes it; the shim works fine with the RStudio HTML pane.
- `results=rd` and install‚Üírender emit both need **4 backslashes** in source.
- Toolchain: use the `/usr/bin` R 4.5 pair; the stray `/usr/local/bin/Rscript`
  (4.6) has its own empty library and can't `R CMD INSTALL`.
